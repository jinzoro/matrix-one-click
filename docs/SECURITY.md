# Security Guide

## Security Model Overview

This stack is designed with a defense-in-depth approach:

1. **Network isolation**: Services communicate on Docker internal networks. Only Traefik is exposed externally.
2. **TLS everywhere**: All external traffic is TLS 1.2+ with strong cipher suites.
3. **Minimal exposure**: Registration disabled by default; admin API requires authentication.
4. **Secret management**: Secrets are generated randomly, stored only in `.env` (gitignored).
5. **Security headers**: HSTS, CSP, X-Frame-Options applied to all HTTPS responses.
6. **TURN SSRF protection**: Coturn denies all private IP ranges.

---

## Network Architecture

```
Internet
    │
    ├── :80  (HTTP → redirected to HTTPS)
    ├── :443 (HTTPS — Traefik)
    ├── :8448 (Matrix federation — Traefik)
    ├── :3478 (TURN/STUN — Coturn, host network)
    └── :5349 (TURN TLS — Coturn, host network)

Docker Networks:
  matrix-proxy (external-facing):
    - traefik
    - synapse
    - element-web
    - well-known

  matrix-backend (internal, NOT accessible from host):
    - postgres      (never exposed to internet)
    - redis         (never exposed to internet)
    - synapse
```

The `matrix-backend` network has `internal: true` in Docker Compose, which means containers on this network cannot reach the internet or the host network. This isolates the database from external access.

## Secret Management

### How secrets are stored

All secrets are stored in `.env` (gitignored). They are:
- Generated by `scripts/generate-secrets.sh` using `openssl rand -base64 48`
- Passed to containers via environment variables
- Never stored in Docker images or config templates

### Secret rotation

To rotate a secret (e.g., after suspected compromise):

**Rotating SYNAPSE_MACAROON_SECRET_KEY:**
```bash
# 1. Generate new secret
NEW_SECRET=$(openssl rand -base64 48)

# 2. Update .env
sed -i "s|^SYNAPSE_MACAROON_SECRET_KEY=.*|SYNAPSE_MACAROON_SECRET_KEY=${NEW_SECRET}|" .env

# 3. Re-process templates
bash scripts/init-synapse.sh

# 4. Restart Synapse
docker compose restart synapse

# Note: This invalidates ALL existing access tokens.
# All users will be logged out and need to log back in.
```

**Rotating POSTGRES_PASSWORD:**
```bash
# 1. Update the password in Postgres
docker compose exec postgres psql -U synapse postgres -c "ALTER USER synapse PASSWORD 'new-strong-password';"

# 2. Update .env
sed -i "s|^POSTGRES_PASSWORD=.*|POSTGRES_PASSWORD=new-strong-password|" .env

# 3. Re-process templates and restart
bash scripts/init-synapse.sh
docker compose restart synapse
```

**Rotating REDIS_PASSWORD:**
```bash
# 1. Update .env
sed -i "s|^REDIS_PASSWORD=.*|REDIS_PASSWORD=new-redis-password|" .env

# 2. Restart both Redis and Synapse
docker compose restart redis synapse
```

### Secrets that must NEVER be changed after first deployment

- `SYNAPSE_MACAROON_SECRET_KEY` — can be changed, but invalidates all access tokens
- The **signing key** (`data/synapse/<server_name>.signing.key`) — changing this means your server loses its identity on the Matrix network. All remote servers that have verified your key will need to re-verify.
- `MATRIX_SERVER_NAME` — cannot be changed. Ever.

## TLS Configuration

Traefik enforces TLS 1.2 minimum with the following cipher suites (configured in `config/traefik/dynamic/tls.yaml`):

```
TLS_ECDHE_ECDSA_WITH_AES_128_GCM_SHA256
TLS_ECDHE_RSA_WITH_AES_128_GCM_SHA256
TLS_ECDHE_ECDSA_WITH_AES_256_GCM_SHA384
TLS_ECDHE_RSA_WITH_AES_256_GCM_SHA384
TLS_ECDHE_ECDSA_WITH_CHACHA20_POLY1305
TLS_ECDHE_RSA_WITH_CHACHA20_POLY1305
```

These are all ECDHE (forward secrecy) cipher suites that score A+ on SSL Labs.

### Testing TLS configuration

```bash
# Test with ssllabs (online)
# https://www.ssllabs.com/ssltest/analyze.html?d=matrix.example.com

# Test with nmap
nmap --script ssl-enum-ciphers -p 443 matrix.example.com

# Test with openssl
openssl s_client -connect matrix.example.com:443 -tls1 2>&1 | grep "Cipher is\|Protocol :"
openssl s_client -connect matrix.example.com:443 -tls1_1 2>&1 | grep "alert"  # Should fail
openssl s_client -connect matrix.example.com:443 -tls1_2 2>&1 | grep "Cipher is"  # Should succeed
```

## Registration Security

Registration is **disabled by default** (`SYNAPSE_ENABLE_REGISTRATION=false`).

To enable registration safely:
1. Only enable if you intend to run a public or semi-public server
2. Consider enabling CAPTCHA verification to prevent bot registrations
3. Consider restricting to specific email domains:
   ```yaml
   # In homeserver.yaml.tpl
   allowed_local_3pids:
     - medium: email
       pattern: ".*@yourdomain\\.com"
   ```
4. Monitor for abuse — spammers will find your registration endpoint

If you want to allow specific people to register without enabling public registration, invite them to use the registration shared secret:
```bash
# Share this command with trusted users (they need shell access or you run it for them)
bash scripts/create-admin-user.sh
```

Or use the [Synapse registration tokens feature](https://element-hq.github.io/synapse/latest/usage/administration/admin_api/registration_tokens.html).

## Rate Limiting

Rate limits are configured in `config/synapse/homeserver.yaml.tpl`. The defaults are conservative and suitable for small/medium deployments:

| Limit | Default | Purpose |
|---|---|---|
| `rc_message.per_second` | 0.2 | Message sending rate |
| `rc_login.address.per_second` | 0.17 | Login attempts per IP |
| `rc_login.failed_attempts.per_second` | 0.17 | Failed login attempts |
| `rc_registration.per_second` | 0.17 | Registration attempts |

Additional rate limiting via Traefik is available in `config/traefik/dynamic/middlewares.yaml` (`synapse-ratelimit` middleware).

## Coturn Security — Preventing SSRF

The TURN server can be exploited to probe internal network services if not properly secured. The `config/coturn/turnserver.conf.tpl` includes a complete set of `denied-peer-ip` rules that block all private, loopback, link-local, and reserved IP ranges.

These rules prevent:
- Port scanning internal Docker networks via TURN
- Accessing the Docker daemon API via TURN
- Accessing cloud metadata APIs (169.254.169.254)
- SSRF attacks targeting internal services

**Do not remove any `denied-peer-ip` lines without understanding the security implications.**

## Admin API Protection

The Synapse Admin API (`/_synapse/admin/`) requires an admin user's access token. It is not separately firewalled but is protected by Synapse's authentication layer.

For additional protection, consider restricting the admin API to specific IP addresses using Traefik middleware:

```yaml
# In config/traefik/dynamic/middlewares.yaml
admin-api-ipwhitelist:
  ipAllowList:
    sourceRange:
      - "203.0.113.10/32"  # Your admin IP
      - "10.0.0.0/8"       # Your VPN network
```

Then attach it to a separate router for `/_synapse/admin/` paths.

## Brute Force Protection

Synapse has built-in rate limiting for login attempts. For additional protection:

### Fail2ban Integration

See the Fail2ban section in [TROUBLESHOOTING.md](TROUBLESHOOTING.md) for setting up Fail2ban with Traefik access logs.

### IP blocking via Traefik

For known bad actors:
```yaml
# In config/traefik/dynamic/middlewares.yaml
block-bad-ips:
  ipAllowList:
    sourceRange:
      - "0.0.0.0/0"
    # Note: Use ipDenyList in Traefik v3 to block specific IPs
```

## Security Headers

All HTTPS responses include the following security headers (configured in `config/traefik/dynamic/middlewares.yaml`):

| Header | Value | Purpose |
|---|---|---|
| `Strict-Transport-Security` | `max-age=63072000; includeSubDomains; preload` | HSTS (2 years) |
| `X-Frame-Options` | `SAMEORIGIN` | Clickjacking protection |
| `X-Content-Type-Options` | `nosniff` | MIME sniffing protection |
| `Content-Security-Policy` | (see middlewares.yaml) | XSS/injection protection |
| `Referrer-Policy` | `strict-origin-when-cross-origin` | Referrer privacy |
| `Server` | (removed) | Hide server software |
| `X-Powered-By` | (removed) | Hide technology stack |

## OIDC / Single Sign-On

To enable OIDC (e.g., with Keycloak, Authentik, or another provider), uncomment the `oidc_providers` section in `config/synapse/homeserver.yaml.tpl` and configure your provider.

Example with Authentik:
```yaml
oidc_providers:
  - idp_id: authentik
    idp_name: "Authentik SSO"
    issuer: "https://auth.example.com/application/o/matrix/"
    client_id: "your-client-id"
    client_secret: "your-client-secret"
    scopes: ["openid", "profile", "email"]
    user_mapping_provider:
      config:
        localpart_template: "{{ user.preferred_username }}"
        display_name_template: "{{ user.name }}"
```

After updating, run `bash scripts/init-synapse.sh && docker compose restart synapse`.

## Monitoring for Abuse

Enable Synapse metrics to monitor for abuse patterns:
```yaml
# In homeserver.yaml.tpl (uncomment)
enable_metrics: true
```

Then add a Prometheus + Grafana stack. Key metrics to monitor:
- `synapse_http_server_requests_received_total` — Request rate
- `synapse_storage_events_persisted_total` — Event creation rate
- `synapse_users_registered_total` — Registration rate (spike = potential abuse)

## Auto-Update Guidance

**We recommend manual upgrades** for this stack. Automated updates for a stateful service like Synapse carry risks:
- Breaking configuration changes
- Database migrations that need monitoring
- Potential data loss if migrations fail

If you want automatic security updates for the base OS, use `unattended-upgrades` (Debian/Ubuntu) with a security-only configuration.

For container image scanning, consider:
- [Trivy](https://github.com/aquasecurity/trivy): `trivy image matrixdotorg/synapse:latest`
- [Grype](https://github.com/anchore/grype): `grype matrixdotorg/synapse:latest`
